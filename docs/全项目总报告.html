第一章 绪论

1.1 本章要解决什么问题

这份总报告讨论的，并不是一个泛泛的“AI 会不会写古诗词”问题，而是一个更具体、也更适合被系统研究的问题：当大模型进入宋词这一高度讲究词牌结构、传统资源池、题材惯性与文体自我约束的文类时，它究竟会表现出怎样的能力边界与风险结构。

如果只从少量样本出发，AI 宋词很容易给人两种彼此冲突的印象：一方面，它似乎已经能写出相当像样、相当有“古意”的作品；另一方面，只要把任务改得更难一些，或者把样本放到更大规模上比较，又会看到格律失稳、主题收缩、来源聚集和内部同质化等一系列问题。因此，本项目真正要做的，不是给出一句笼统的“能”或“不能”，而是把这种复杂表现拆成若干可比较、可复核的问题层。

1.2 本项目真正关心的四类问题

为了避免把不同层面的现象混成一句“写得好不好”，本项目把核心问题拆成四条主线。表 1 给出的是本研究真正要回答的问题框架。

1.3 研究对象：六类 AI 任务与一个真实宋词库

本项目的实验对象不是单一模型，也不是单一 Prompt，而是由六个 AI 数据库和一个真实宋词数据库共同构成的比较体系。六个 AI 数据库对应 a1 / b1 / b2 / b3 / c1 / c2 六类任务条件，总计 3122 首有效 AI 宋词；真实宋词库则收录 21053 首宋词，用作历史基线和来源参照。

这套设计的关键，不在于“样本多”，而在于任务之间具有明确梯度。a1 是自由生成，b1 加入词牌约束，b2/b3 在词牌约束外再加入主题，c1/c2 则把模型推入固定《沁园春》的长调高压测试。换言之，这不是六个彼此孤立的实验池，而是一条从相对自由到高度加压的连续链条。

在模型层面，本项目最适合做稳定横向比较的是三家主模型：doubao-seed-2.0-pro、deepseek-3.2 和 qwen3.6-plus。其余模型更多承担补充观察、边缘参照或探索性试样的功能。这一处理并不是为了简化问题，而是为了保证后文讨论时，不会把小样本探索性条目与核心主模型强行放在同一解释强度上。

1.5 本章小结

本项目讨论的核心，并不是“AI 写的宋词漂不漂亮”，而是它在宋词这一高约束文体里究竟如何表现出形式能力、内容惯性、来源关系与内部收缩。

也正因为如此，后文的重点不会放在个别惊艳样本或个别失败样本上，而会尽量通过任务梯度、数据库比较、模型比较和样本册对照，把这些能力与风险放到同一套可复核框架里讨论。只有这样，整份报告最后给出的结论，才不会停留在印象判断，而能真正服务于后续的生成设计、筛选和研究延伸。

第二章 项目设计与数据体系

2.1 本章要解决什么问题

这一章不直接进入“哪个模型更好”“哪组结果更高”，而是先回答三个更基础的问题：第一，本项目到底设计了哪些任务条件；第二，这些任务条件之间是什么关系，而不是彼此毫无关联的六个数据库；第三，AI 生成库与真实宋词库在整个研究中各自承担什么角色。

从整体上看，本项目不是把若干模型随意丢进若干 Prompt 里做散点式采样，而是搭出了一条逐层加压的实验梯度。模型先处于自由生成状态，随后被加入词牌约束，再被加入主题约束，最后被推入《沁园春》这一长调高压测试。也就是说，a1 到 c2 的关系更接近“条件递进链”，而不是“六个平行池子”。

因此，本项目真正观察的是：当外部约束逐步增强时，大模型的形式控制、主题组织、内容收缩与后续风险结构会如何发生系统变化。

generate/prompts.py 中的五类 Prompt 可以概括为：

  1. type_c：允许模型自行决定词牌，要求输出合法 JSON，并强调写出完整宋词。
  2. type_b：由外部提供词牌名，要求依给定词牌创作，但不规定主题。
  3. type_d：同时给定词牌和主题背景，要求模型在明确情境内完成创作。
  4. type_a：固定《沁园春》，并直接给出较完整的格律骨架，是显式模板式强结构任务。
  5. type_e：同样固定《沁园春》，但不再提供完整词谱，只以简短文字提醒模型遵守词林正韵与钦定词谱。

这里必须特别区分 c1 与 c2。它们都固定为《沁园春》，但并不是同一种强约束。c1 的核心在于“给模板”，c2 的核心在于“给提醒”。因此，后续若出现显著差异，不能简单归因于主题不同，更应理解为 Prompt 结构本身就在改变模型的生成方式。

本项目最重要的两类主题不是抽象标签，而是完整叙事情境。为了避免后文只剩下“改革”“玉兰”这样的简写，这里将其原始主题背景完整列出：

改革主题原始文本：

假设你是一个修仙世界一个宗派的年轻长老，看到了宗派中传统的规矩和观念对宗派中的弟子的压抑，希望能改变这个现实，但是发现严格的规矩的确能一定程度上能保证在危机面前的战斗力，而且宗派中很多长老也很反对这个改革。你在挣扎中写了一首词来抒发自己的想法。

玉兰主题原始文本：

假设你是一个喜欢写宋词的大学生，最近在学业和面试上屡遇挫折，但是你看到了一颗盛开着白玉兰花的树，似乎重新找到了生命的实在与价值，便写了一首词。

这两类主题的差异并不只是“题材不同”。改革主题带有更明显的制度冲突、身份矛盾与价值判断，因此更像一条复杂叙事线；玉兰主题则更容易落到“受挫 - 观物 - 转念”的单线抒情路径中。这个差异会在后续的内容分布、来源风险与同质化分析中持续出现。

2.4 AI 数据库与真实宋词库的关系

本项目共有六个 AI 生成数据库，总样本数为 3122 首；另有一个真实宋词数据库 real_song_ci_dataset.db，共收录 21053 首真实宋词。两者在研究中的功能并不相同。

第一，六个 AI 数据库承担的是“实验对象库”的角色。后续关于格律能力、意象分布、主题类别、真实来源相似性和内部同质化的大部分分析，都是围绕这六个库展开。

第二，真实宋词库承担的是“历史基线库”和“参照库”的角色。它既不是简单的陪衬，也不是单纯用来证明 AI 与古人“像不像”。在真实来源相似性分析中，它提供整首与句子两级的历史比对对象；在同词牌历史基线分析中，它又帮助我们判断：某种高相似究竟是异常集中，还是该词牌传统内部本来就高度接近。

第三，真实来源分析的运行元数据表明，本项目在该部分共处理 3122 首 AI 词、49304 个 AI 句段，对照 21053 首真实宋词、270107 个真实句段；在目标词牌层面，则涉及 36 个重点词牌。这说明真实语料并不是只在最终结论处被零散调用，而是作为全量比较基线被系统接入。

2.6 本章小结

本项目的数据体系有三个最核心的特征。第一，它是一条逐层增强约束的实验链，而不是六个平行池。第二，Prompt 差异不只是“有没有主题”，还包括“是否给显式格律模板”“是否固定到长调词牌”等结构差异。第三，真正稳定支撑横向对比的，是三家主模型与六个 AI 库构成的主体实验面，而真实宋词库则提供历史基线与来源参照。

换句话说，后文所有结果都必须回到这里来解释：我们比较的从来不只是“模型 A 对模型 B”，而是“模型在不同约束结构下会呈现出怎样的能力和风险变化”。

7 已入库样本的 content 后续由 generate/extract_imagery.py 批量抽取意象与类别 补全 imagery 与 category
8 多库样本汇总 统计脚本统一归并模型名、聚合错误类型和分数分布 各类分析 CSV 结果

从程序实现上看，生成主链路主要位于 generate/generator.py 中。该脚本先通过 load_tasks() 读取任务配置，并把 provider 下的 API 信息写回到每个任务对象中；然后由 get_prompt() 根据任务类型决定是否填入词牌和主题；再使用异步并发方式把请求发送到模型接口。

模型返回结果后，parse_json_response() 负责进行第一次“轻清洗”。这里的清洗并不是文学意义上的润色，而是技术意义上的格式整理：它会删除模型思考标签，尝试在整段文本中抓取合法 JSON 对象，并将其解析为结构化字段。因此，数据库中的 content 不是“原始裸文本”，而是“从原始响应中成功抽出的 JSON 正文”。

3.3 任务配置文件的组织方式

当前项目中至少存在两类任务配置来源：

  1. generate/tasks.json：当前或最近一次运行所使用的活动配置。
  2. generate/json/1.json 至 generate/json/6.json：保存历史任务快照，用来回看不同阶段实验的具体配置。

这些配置文件虽然内容不完全一致，但组织方式相近，通常都包含以下字段：

  • provider：调用哪一家接口服务。
  • model：底层大模型名称。
  • prompt_name：对应 prompts.py 中哪一种 Prompt 模板。
  • db_name：结果写入哪个 SQLite 数据库。
  • total_requests：计划请求总数。
  • max_concurrency：并发上限。
  • request_interval：请求间隔。
  • cipai_pool：若任务要求从多个词牌中随机抽取，则在此提供候选池。
  • theme：若任务要求围绕某个情境创作，则在此提供主题背景文本。

这里要特别强调一点：任务配置中的请求数只是计划值，不是结果值。只有通过 JSON 解析、正文提取、格律评估并最终成功写入数据库的样本，才会进入后续统计。因此，理论请求数、原始调用数与有效入库数三者并不总是相同。

3.5 原始返回值、清洗文本与后续分析字段之间的关系

项目中的文本链路可以概括为：

raw_response -> JSON 解析 -> content -> 格律评估 -> rhythm_score/rhythm_errors -> 意象提取 -> imagery/category

这条链路中有三个关键转换。

第一，原始返回值到正文文本的转换。模型返回往往不仅包含正文，还可能夹带思考标签、说明文字、代码围栏甚至格式噪声。parse_json_response() 的职责就是从这些噪声中尽量抽出合法 JSON，并保留其中的 content 作为后续唯一使用的正文文本。

第二，正文文本到格律结果的转换。SongCiEngine.evaluate() 并不直接读取 raw_response，而是只针对 content 与 cipai 做校验。换言之，格律分数是正文层面的程序评估结果，而不是模型自报结果。

第三，正文文本到意象/类别字段的转换。extract_imagery.py 并不依赖格律字段，而是直接把正文再次送入一个专门的抽取 Prompt，要求输出意象列表和九类题材之一，再把结果写回数据库。

因此，一个样本在数据库中的完整形态，其实是“原始生成 + 程序格律评估 + 二次语义抽取”三条流程叠加出来的结果。

真实宋词数据库 real_song_ci_dataset.db 使用 real_ci_data 表，实际字段包括：

  • id：主键。
  • author：作者名。
  • full_rhythmic：完整格律信息。
  • cipai：词牌名。
  • content：正文。
  • rhythm_score：格律分。
  • rhythm_errors：格律错误说明。

可以看到，真实库并不只是一个“原文文本仓库”，它也已经带有格律相关字段。这为后续真实来源分析和历史基线比较提供了更细的支撑：既可以按正文做相似性检索，也可以按词牌和格律层面进行辅助判断。

3.8 数据结构上的几个方法论提醒

第一，数据库中的 content 是“成功结构化后的正文”，而不是所有原始返回的自然全集。因此，当报告统计样本数时，它统计的是“有效样本”，而不是所有调用尝试。

第二，imagery 与 category 具有后处理性质。它们很适合用于宏观分布比较，但不应被误当作人工标注金标准。

第三，rhythm_errors 虽然在数据库中以 JSON 字符串保存，但后续统计脚本会把它拆成具体错误项，并进一步归为“字数不符”“应平实仄”“应仄实平”“出韵/押韵问题”“未收录字警告”“词牌变体智能纠偏”等标准类型。也就是说，数据库原始字段是长格式错误清单，结果 CSV 则是聚合后的宽表统计。

3.9 本章小结

本项目的数据并不是“一次生成，一次入库，直接分析”的简单流水线，而是一条分阶段加工的研究链。第一阶段生成并解析正文，第二阶段计算格律，第三阶段补充意象和类别，第四阶段再统一模型口径、聚合错误与输出统计表。

理解这一点非常关键，因为它决定了后文所有数字的性质：有的是原始字段，有的是程序评分，有的是模型辅助抽取后的标签，有的是进一步归并聚合后的统计结果。只有把这些层次区分清楚，总报告中的方法和结论才会真正站得住。

第四章 格律分析方法

4.1 本章要解决什么问题

本项目最基础、也最不能含糊的方法环节，就是格律分析。因为后文无论讨论模型差异、词牌难度，还是讨论强约束为什么会失败，最后都要回到一个前提：这里所说的“格律分”究竟是怎么计算出来的。

这一章要解决的不是泛泛而谈的“机器也会检查平仄”，而是具体说明：程序依赖哪些词谱和韵书，如何逐字判定字数、平仄、押韵，如何处理多音字，如何定义零分样本、高质量样本与自动纠偏。

4.2 词谱、平水韵与词林正韵的角色分工

格律分析的核心引擎是 engine/songci_engine.py。它在初始化时同时加载三类基础资源：

  1. engine/词谱.json：词牌结构库。
  2. engine/韵书.json：平水韵字库。
  3. engine/词林正韵.json：词林正韵分部表。

这三者的功能并不重复。

词谱.json 解决的是“这一词牌应当写成什么样”的问题。根据实际统计，该文件当前包含 818 个词牌条目、2475 种格式变体。也就是说，引擎并不是只会校验少数常见词牌，而是在较大的词牌全集上进行匹配。

韵书.json 解决的是“某个字属于平、上、去、入哪一类韵部”的问题。程序会先把平水韵条目拆成按声调组织的字集，并建立“韵目名称 -> 字集”的映射，供后续严格押韵判断使用。

词林正韵.json 解决的是“若严格平水韵不完全命中，是否还能在词林正韵的更大韵部内通押”的问题。也就是说，平水韵负责窄韵判定，词林正韵提供宽韵兼容。这一层设计让引擎既有明确边界，又不至于在传统可通押情形下过度机械。

4.3 引擎的总体工作流程

表 10 格律分析流程表

步骤 程序动作 方法含义
1 接收 content 与 cipai 以正文和词牌为输入开始评估
2 根据词牌名寻找候选词牌家族 允许令、引、近、慢、减字、摊破等相关变体进入候选
3 对每个候选格式逐一校验 逐字比较字数、平仄与押韵
4 为每个候选格式计算得分 根据错误类型累积扣分
5 选取得分最高的匹配格式 若分数相同，则优先选字数差更小、错误更少的变体
6 返回最佳结果与错误摘要 输出 score、summary、matchedVariant 等结果

这里最值得注意的是第二步和第五步。引擎不是死板地只看一个词牌名对应的一个固定格式，而是先用 _get_candidate_patterns() 做词牌家族智能路由。它会去掉“减字、摊破、偷声、令、引、近、慢、犯”等前后缀，尝试把名称归到同一核心词牌族中，再在候选变体里寻找最佳匹配。

这种做法的好处是：当模型写出的字数和句式更接近某个相关变体，而不是它自报的词牌名时，引擎仍能给出最接近实际情况的评分，而不会直接因为名称不完全一致而报废。

4.4 字数、平仄、韵脚与出韵的具体判定

4.4.1 字数判定

程序首先用正则表达式提取正文中的汉字，忽略标点和其他非汉字字符。随后，它将实际字数与词谱格式要求的字数做直接比较。

这一条是最严格的硬门槛：若实际字数与要求字数不一致，引擎立即返回 score = 0，并在错误摘要中给出“该变体要求多少字、实际多少字、相差多少字”的说明。换言之，字数不符不是普通扣分项，而是直接触发零分。

4.4.2 平仄判定

在非韵脚位置，引擎只检查当前字是否满足该位置对平仄的要求。

  • 若规则要求平声，而程序判定该字为仄声，则记为“应平实仄”。
  • 若规则要求仄声，而程序判定该字为平声，则记为“应仄实平”。

每出现一次这类错误，累积罚分 3。由于总分最终按 100 - total_penalty * 4 计算，因此每个平仄错误对应实际分数下降 12 分。

从结果统计看，这两类错误是项目中最主要的格律瓶颈，尤其在《沁园春》这类长调条件下最为突出。也就是说，大模型最难维持的不是“能不能写出某个句子”，而是能否在较长结构中稳定把每一个位置都写对。

4.4.3 韵脚判定

在韵脚位置，引擎会先判断该处应押平韵还是仄韵，再根据多音字候选、首韵槽位和已有韵部信息选择最合适的读法。

程序对韵脚的处理分三层：

  1. 优先尝试严格命中平水韵中的同韵部字集。
  2. 若无法严格命中，则尝试在词林正韵的同一大韵部内通押。
  3. 若前两步都不成立，则退回到只满足平仄要求的保底读法。

如果韵脚连平仄都不符合，会记为韵脚平仄错误，罚分 4；如果平仄虽然对，但不押前文已确立的韵部，会记为“出韵”，罚分 5。这意味着韵脚错误在评分体系中权重高于普通平仄错误。

4.4.4 多音字判定

多音字不是简单地“随机选一个读音”。程序会结合当前位置应平还是应仄、该处是否是韵脚、当前已建立的押韵槽位等信息，优先选取最符合格律要求的读法。若最终使用的是多音字特定读法，错误摘要中会标注“多音字，取某声读”，以保留解释路径。

4.4.5 未收录字判定

若某字在平水韵字库中未收录，引擎会将其记为警告，但不扣分。这类情况会被写入错误摘要，并在后续统计中归为“未收录字警告”。

这一设计说明，引擎在方法上区分了“规则违背”和“资源缺失”。后者被保留为不确定性提示，而不直接作为格律失败。

程序的最终得分公式为：

此外，引擎将 score >= 60 视为基本有效样本；而统计脚本进一步把所有样本分成五个区间：

  • 100 分：完全满分样本。
  • 80-99 分：高质量样本。
  • 60-79 分：中等样本。
  • 1-59 分：低质量样本。
  • 0 分：零分样本。

这里的“高质量样本”并不是文学评价意义上的“写得好”，而是方法意义上的“格律层面较稳定”。同样，“零分样本”也不是说内容一无可取，而是说它在字数这一最硬的结构门槛上已经失守。

4.6 自动纠偏与词牌变体匹配

本项目格律评估中一个很有特色的机制，是“词牌变体智能纠偏”。

如果模型自报的词牌名与实际写出的字数/格律更接近另一相关词牌变体，引擎不会简单判为彻底失败，而是会把“实际匹配的词牌”写入 matchedVariant，并在错误摘要最前面插入提示信息，例如：该词实际更匹配某个相关变体，而非模型自报的词牌。

这一机制在后续统计中被单独归类为 variant_autocorrect，并明确不计入“实质性错误数”。统计脚本还将“无实质性错误”定义为：除自动纠偏之外，其他错误类型均为零。

换句话说，自动纠偏的意义不是替模型“洗白”，而是把“词牌名说错但结构接近某变体”与“真正的平仄、字数、押韵错误”区分开来。对于宋词这种本身就存在大量词牌家族和变体形式的文本系统，这一层区分是必要的。

4.8 方法边界与阅读提醒

本项目格律引擎的强项是结构性校验，而不是文学性判断。它能告诉我们一首词在字数、平仄和押韵上哪里出错，但不会判断它是否“有意境”“有新意”。

同时，这套方法也有三个边界需要在总报告中明确说明。

第一，它是字符级、位置级的格律校验，而不是人工词学批评。程序能高效而稳定地筛查结构问题，但不能替代人工细读。

第二，多音字与宽韵兼容让引擎具备现实可用性，但也意味着部分边界样本带有“最佳可解释匹配”的色彩。换言之，分数仍是严格程序化结果，但对少数边界样本应允许人工复核。

第三，长调任务尤其是《沁园春》会显著放大所有细粒度声律错误。后文如果发现 c1/c2 分数显著走低，并不代表模型完全不会作词，而是说明它在高字数、高结构密度场景下难以稳定维持局部声律规则。

4.9 本章小结

本项目的格律分数并不是模糊印象分，而是由词谱、平水韵和词林正韵共同支撑的程序化评分结果。其核心逻辑可以概括为：字数先行，平仄为主，押韵加严，变体纠偏单列，多音字择优解释。

这套方法为后文提供了一个非常关键的基础：我们讨论“模型格律能力”时，说的不是主观感觉，而是以同一套词谱、同一套韵书和同一套扣分规则对所有样本做出的可复核比较。

第五章 格律结果总览

5.1 本章要解决什么问题

如果说第四章回答的是“格律分是怎么计算出来的”，那么第五章要回答的就是：在同一套评分规则下，不同任务条件、不同模型与不同词牌究竟表现出了怎样的稳定差异。

这一章最重要的，不是简单列出哪个数据库分高、哪个数据库分低，而是把三个层次区分清楚。第一，任务层面上，什么样的约束结构最容易让模型守住格律，什么样的约束结构会使格律迅速塌陷；第二，模型层面上，谁在短调、长调、强主题与显式模板下更能维持结构控制；第三，词牌层面上，难度究竟来自字数、句法长度，还是来自长调内部细粒度声律规则持续维持的困难。

从全项目结果看，格律并不是一个边缘指标，而是所有比较中最先拉开差距的一道门槛。它最清楚地揭示了一个事实：大模型不是完全不会写“像宋词的句子”，而是当结构被拉长、主题被压紧、局部位置越来越多时，最难维持的是整首词内部连续而稳定的平仄与押韵。

5.6 为什么长调更难，为什么主题会拖累形式控制

综合前面几组结果，可以把本项目的格律压力机制概括成两条线。

第一条线是结构线。词牌越长，句段越多，模型需要持续维持正确的局部位置就越多，任何一次轻微偏离都更容易在后文不断累积。所以《沁园春》之类长调并不是“平均每个位置都更难”，而是它把小错误积少成多的风险放大成了系统性失败。

第二条线是内容线。主题一旦加入，模型必须同时处理情境设定、叙事路径与词牌格局，这会直接分流它对形式约束的注意力。因此 b1 -> b2/b3 的下滑并不是偶然波动，而是主题竞争了本该用于形式控制的生成资源。

最值得强调的是，这两条线在 c2 身上叠加到了极限：它既是长调，又是强主题，又没有 c1 那样的显式格律模板。正因为如此，c2 不只是“稍微难一点”，而是把大模型的格律控制能力真正压到了底线附近。

5.7 本章小结

格律结果最清楚地说明，本项目中的形式控制并不是均匀分布的能力，而是会随着约束结构迅速分化。固定词牌而不给主题，是目前最利于模型守住格律的任务形式；一旦主题加入，格律就开始明显下滑；当任务进一步进入《沁园春》这类长调高压场景时，模型差距会被最大幅度放大。

这也意味着，后文所有关于主题、相似性与同质化的讨论，都不能脱离第五章的前提：很多“内容风险”之所以会出现，首先是因为模型在高压任务中已经很难把形式稳稳地守住了。

第六章 意象分析与内容分布

6.1 本章要解决什么问题

第五章讨论的是形式控制，第六章则转向内容组织本身。这里要回答的问题不是“模型写得像不像词”，而是“模型通常在写什么”“不同条件下最容易调动哪些意象”“主题是怎样把内容空间压缩成少数稳定路径的”。

这一章尤其重要，因为宋词创作的差异并不只体现在句式层面，也体现在意象系统和题材归类上。即使两首词在字面上并不完全相同，只要它们总是在调用同一组景物、同一组人物关系、同一种抒情路线，就已经形成了一种更深层的内容收缩。后文的内部同质化分析，实际上就是建立在这一章的描述性统计基础之上的。

6.8 本章小结

内容分布结果表明，AI 宋词并不存在一个真正均匀开放的“创作空间”。在无主题条件下，它会自动回到传统宋词最熟悉的离愁、旅思与秋景资源池；在给定主题后，它又会迅速收缩到少数稳定的题材与意象链上。

其中，改革主题更容易形成“宗门制度化”的语义场，玉兰主题则更容易把作品压缩到“花木寄托 + 困顿自励”的单一路径。模型之间当然仍有风格差别，但这种差别并不足以抵消总体收缩趋势。也正因如此，后文讨论内部同质化时，不能只看文本表面是否重复，还要看到意象池和题材分布本身已经在变窄。

第八章 真实来源相似性分析

8.1 本章要解决什么问题

这一章要回答的，并不是一个简单的“AI 有没有抄袭”的二元问题，而是一个更适合本项目的方法问题：AI 宋词与真实宋词之间的相似，究竟属于传统化用、共享资源池、局部模板复用，还是已经接近需要人工重点复核的来源聚集。

之所以必须这样分层讨论，是因为宋词本身就是一个高度共享修辞资源、意象资源与短语资源的文类。如果不先建立真实宋词内部的历史基线，很多本来属于传统共相的相似现象，会被误读为异常借用；反过来，如果只看“高相似句很多”，又会把大量短句套语的正常重合误判成高风险证据。

8.8 本章小结

真实来源相似性分析最重要的结论，不是“AI 没有风险”，也不是“AI 大量抄袭”，而是必须把风险拆解到更合适的层次来讨论。

整体而言，绝大多数 AI 宋词更像是分散化用、局部语块复用和传统资源池重组，而不是整首级单一来源重写；但少数高风险样本确实存在，而且它们会集中出现在特定模型、特定词牌与特定任务条件中。正因为如此，本项目后续更适合继续使用“借用关系”“来源聚集”“局部改写风险”这类表述，而不是一开始就用过于刚性的“抄袭”判断。

第九章 内部同质化分析

9.1 本章要解决什么问题

第八章处理的是“AI 与真实宋词之间像不像”，第九章则处理另一个同样关键的问题：AI 样本彼此之间会不会越来越像。

这个问题之所以必须单列出来，是因为即使某些样本与真实宋词并不构成高风险来源借用，AI 内部仍然可能出现另一种问题：它们未必都像某一首古人作品，但却越来越像彼此，越来越依赖同一套抒情骨架、句法模板和意象资源。这种内部同质化不会直接表现为“对真人文本的单点复制”，却会显著削弱模型输出的多样性与创造性。

9.2 同质化的三层结构与两个综合指标

表 35 同质化指标释义表

指标 所在层次 含义

其中，whole_score 更接近“整体立意和情境结构越来越相似”，sent_score 更接近“局部句法骨架和表达模板越来越接近”，imagery_score 则说明作品是否在反复调用同一套景物、人物关系和题材标签。core_mihi 与 mihi_full 则分别承担“稳健比较”和“完整解释”两种功能。

9.9 本章小结

内部同质化分析表明，AI 宋词的问题并不只在于偶尔和真实宋词太像，更在于随着约束层层增强，它们彼此之间也越来越像。

这种“越来越像”首先表现为整首词在写法与情境结构上的收缩，其次才可能逐步表现为句法模板与意象系统的重复。玉兰主题比改革主题更容易诱发这种收缩；《沁园春》比其他核心词牌更容易把收缩推到高位；三家主模型都存在超额同质化，但收缩机制并不相同。至此，我们已经可以把本项目的核心风险结构初步勾勒出来：形式压力、主题压力、来源聚集与内部同质化，并不是互不相干的四件事，而是在同一组高压任务中不断相互强化的。

10.3 “稳而窄”的代表：c1:1《阶前见玉兰感作》

前文已经指出，doubao-seed-2.0-pro 在高压任务里经常同时呈现“格律最稳”和“总体同质化最高”这两个看似矛盾的特征。c1:1《阶前见玉兰感作》 正是这种状态的代表：它的 rhythm_score = 100，类别为 咏物寄托，整首来源分位却都不高，说明它并不是通过贴近某一首真实词来获得稳定性，而是通过一条已经非常成熟的写作路径把作品稳稳组织起来。

这首词的路径非常清楚：开篇是“案积尘笺，笔冷清灯，屡负选场”的受挫场景，中段转到“万朵冰蕤压矮墙”“落清芬满袖”的玉兰发现，结尾再落回“初心未改”“再整行装”的自励收束。单看这一首，仍可视为完成度很高的规范写法；但只要把它和同库同模型的 c1:2《校中见玉兰作》、c1:3《寓舍见玉兰有感》 放在一起，就会发现三首满分样本几乎共享同一骨架：都是先写案头困顿或名场失意，再写阶前见花，最后回到“再谱新章”“再试锋芒”一类重新出发的尾声。

这类样本最能说明第九章所说的“高质量但偏稳态收缩”。它的问题不在于单篇粗糙，而在于模型把玉兰主题过于稳定地翻译成了“受挫青年 - 偶见白花 - 恢复志气”的标准流程；形式越稳，这条成熟路径越容易被反复调用。

10.4 “主题压过形式”的代表：c2:201《玄渊阁规谏》

如果说上一例展示的是“形式稳住之后内容如何收缩”，那么 c2:201《玄渊阁规谏》 则展示了另一种相反形态：这首词的 rhythm_score = 0，但主题表达其实非常完整。它被归到 豪情壮志 类，正文里连续出现“铁律千重锁玉京”“欲破陈规”“阵图星列”“夜阑独对残灯”“待春雷动，看新篁破土，再立天衡”这一整套改革叙事链条，几乎把 c2 的制度冲突、守旧阻力与改革愿望都明白写了出来。

也就是说，这首词真正失守的不是“写不出主题”，而是“在长调和强约束条件里，主题推进挤占了局部格律控制”。从文本阅读上看，它很容易让人感到主题鲜明、情绪也足够充足；但从方法层面看，它恰恰说明第六章与第五章的两个结论可以同时成立：模型确实抓住了改革主题的高压语义场，却没有能力把这套表达稳定安放进《沁园春》的完整结构之中。

这种案例提醒我们，格律失败不等于内容失败。相反，在某些高压任务里，模型甚至可能先把“要说什么”说得非常清楚，再在“怎样按词体说出来”这一步骤上整体塌陷。

10.5 “高相似不等于高风险”的代表：a1:81《秋江夜泊》

第八章最容易被误读的一点，是把“整首很像传统宋词”直接等同于“来源风险很高”。a1:81《秋江夜泊》 正好可以用来纠正这种直觉。它的 whole_sem_percentile = 0.9781，已经进入历史高位尾部；但语义和字面的主来源覆盖率都只有 0.0833，说明它虽然整体气息极像传统宋词，却并没有大面积回到同一首来源上。

这首《青玉案》里，“西风”“芦花雪”“孤舟”“寒汀”“渔笛”“霜露”“雁背斜阳”几乎把经典羁旅词的核心资源都调动了出来，因此整首会显得异常熟悉。这种熟悉感是真实的，但它更接近高密度地调用传统资源池，而不是向某一首现成作品集中回写。换言之，它像的是一整类成熟词境，而不是一首单一原作。

这正是第八章反复强调的“分散化用”案例：文本传统感很强，文学上甚至可能相当完整，但风险判断不能只看“像不像”，还要看它究竟像得集中不集中。

10.9 本章小结

案例分析把前九章的抽象结论重新压回到了样本层。回到具体文本之后，可以更清楚地看到：有些样本的问题是“太稳”，有些是“主题太强以至于压垮形式”，有些虽然整体非常像传统宋词，却仍属于分散化用；真正需要警惕的，则是那些在多个关键句位持续回指同一来源的样本。

更重要的是，形式压力、主题压力、来源聚集与内部同质化并不是各自孤立出现的。它们常常在同一批高压任务里同时发生，只是有的先表现为格律失稳，有的先表现为内容收缩，有的最后才进入来源复核区间。第十章因此提供的不是几个零散趣味样本，而是一组能够反过来验证整份报告主线判断的文本证据。

第十一章 综合讨论

11.1 如何整体理解“大模型宋词生成能力”

如果把前十章的结果放在一起看，本项目最重要的判断并不是“AI 会写”或“AI 不会写”这样简单的二元结论，而是：大模型已经能够在相当程度上进入宋词这一文体的外观、语气和局部组织方式，但它的能力并不均衡，而且这种不均衡会随着任务约束增强迅速暴露出来。

在较宽松条件下，模型往往可以写出表面上相当像样的宋词文本；但一旦同时叠加词牌压力、主题压力和长调压力，它们就会出现明显分化。有的模型能把格律稳住，却更容易回到成熟、可复用的写法；有的模型对主题响应更强，却也更容易在个别任务里快速收紧到少数文本骨架；还有的模型看上去变化更大，但并不意味着它就真正更自由，而可能只是以另一种方式退回到固定意象与固定情境上。

11.2 格律能力、内容能力、来源风险与同质化之间是什么关系

前文已经反复显示，这四类现象并不是互相分开的。它们更像同一组压力在不同层面的投影：当任务要求上升时，模型首先会在整体情境和内容路径上收缩；随后格律控制开始承压；再往后，在某些模型和某些词牌上，就会出现局部来源聚集与更明显的内部同质化。

其中最容易造成误判的，是把“像宋词”“格律高”“风险低”“同质化低”当成自动绑定的一组属性。事实上，本项目看到的情况恰好更复杂：格律高并不保证来源分散，像宋词也不保证内容多样，而同质化高也不必然意味着整首都像某一首古人作品。也正因为如此，第十一章需要把三家主模型的综合结构重新放到一张表里看。

这张表最值得强调的一点，是“优势”与“风险”常常来自同一能力。doubao 的优势来自更强的稳定组织能力，而这恰恰也是它容易形成稳态收缩的原因；deepseek 的主题表现力更强，而这也正是它更容易在特定任务里快速靠近少数文本骨架的原因；qwen 的变体看起来更多，但在真正高压场景里，它也并没有自动逃出收缩，只是收缩形态与前两者不同。

11.3 为什么“更像宋词”不一定意味着“更好”

在阅读 AI 宋词时，最自然的直觉往往是：越像传统宋词，质量就越高。但本项目的结果提醒我们，这种判断只能成立一半。因为“像”至少有三种不同来源：它可能来自真正较强的词体组织能力，也可能来自对传统资源池的熟练调用，还可能来自对成熟句法模板和现成骨架的依赖。

例如，第十章里的 a1:81《秋江夜泊》 整首语义分位极高，却仍然更接近分散化用；c1 下的 doubao 玉兰样本格律和表达都很稳，却也同时表现出明显的路径重复；而 deepseek 的若干高风险样本则说明，有些文本之所以“像”，并不是因为它们整体文学完成度更高，而是因为关键句位连续回到了同一来源。也就是说，“像宋词”本身是一个需要继续拆解的表面现象，而不是结论本身。

11.4 为什么“更稳”往往伴随“更收缩”

这一点几乎是整份报告最重要的方法论发现之一。高压任务之所以难，并不是因为模型突然“不知道写什么”，而是因为它必须在有限生成预算里同时满足太多条件：词牌、平仄、押韵、主题背景、情绪推进、古典措辞与文体感。一旦这些压力叠加，模型最容易采取的策略，往往不是现场开辟一条新路，而是回到自己最熟悉、最可控、最容易成功的那几条成熟路线。

从生成角度看，这种收缩几乎是理性的。它能显著提高完成概率，也更容易保证文本看起来像一首“合格的词”。但从研究和使用角度看，它又意味着另外两层代价：第一，输出之间会越来越像；第二，模型更容易被少数传统文本骨架和少数题材资源池牵着走。也正因为如此，本项目里“更稳”往往不是单纯的优点，而是一种需要结合多样性和风险一起评估的能力。

11.5 为什么不同模型会走向不同的风险结构

从结果上看，三家主模型并没有简单排成一条“谁更强谁更弱”的线性序列，而更像是走向了三种不同的风险结构。doubao 的问题是规范化过强，deepseek 的问题是主题牵引过强，qwen 的问题则更接近在高压条件下仍然缺乏足够稳固的长调组织能力。

这意味着，后续对模型的使用和评价也不应只看一个总分。若目标是高完成度展示，可能更看重格律稳定和整体顺畅；若目标是研究主题生成，可能更看重模型是否会被特定语义场快速牵引；若目标是批量生成再筛选，则又必须把来源风险和同质化风险一起放进质量控制流程中。模型之间真正不同的，不只是“会不会写”，而是“各自会在哪个环节先失守”。

综合来看，大模型宋词生成已经具有相当强的文体模仿能力，但这种能力不是均匀展开的，而是建立在一组非常具体的优势与代价之上。它越能稳定地产生“像样”的词，往往也越可能回到少数成熟路径；它越能快速响应主题，也越容易在特定场景中发生突出的局部来源聚集与内容收缩。

因此，本项目最终给出的不是单一性能排名，而是一组更适合真实使用的结构性判断：模型并非只是“强”或“弱”，而是在不同压力条件下以不同方式表现出自己的能力边界。

第十二章 结论与应用建议

12.1 项目的最终核心结论

经过前面十章的系统分析，本项目可以得出一个相对稳妥的总判断：大模型已经能够写出看上去相当像宋词、局部甚至颇具完成度的文本，但在高约束条件下，它们仍然会明显暴露出形式控制、内容收缩、来源聚集与内部同质化等结构性问题。

更具体地说，这些问题并不是平均均匀分布在所有模型和所有任务里，而是高度依赖任务类型、词牌难度、主题形式和模型自身的生成偏好。也正因为如此，后续若要真正使用 AI 生成宋词，最有意义的不是问“哪个模型绝对最好”，而是问“在什么任务下、为了什么目标、需要防什么风险”。

12.2 如果想让 AI 写得更稳，该怎么设计任务

若目标是尽量提高成稿率和形式完成度，本项目给出的最明确建议是：适当增加结构扶手，而不是单纯提高主题压力。c1 与 c2 的差异已经说明，同样固定《沁园春》，给出更明确的格律骨架会显著改善结果；而只用简略提醒，则很容易让模型在长调里整体失稳。

从实际操作角度看，更稳妥的做法包括三点：第一，优先选择模型已经较熟悉、项目中也验证过相对稳健的词牌；第二，把任务拆成“先确定词牌和基调，再写完整词”的较清晰指令；第三，在高压任务里加入适量结构示意，而不是完全把格律细节交给模型自由处理。换言之，如果目标是稳定，不应迷信“纯自由发挥”，而应为模型提供必要但不过度拥挤的结构支架。

12.3 如果想让 AI 写得更多样，该怎么设计任务

若目标是尽量避免所有作品都走向同一条成熟路径，则必须反过来压制模板收缩。最重要的不是一味删掉约束，而是避免把任务设计成“只有一条最容易成功的路”。例如，玉兰主题之所以特别容易收缩，是因为它天然鼓励“受挫 - 见花 - 转念”这一单线叙事；若想提升多样性，就应在主题描述中明确开放多种情境、身份位置和情绪走向，而不是只给出单一路径暗示。

此外，应尽量避免批量生成时始终使用同一词牌、同一主题、同一指令强度。词牌轮换、主题层次变化、适当引入角色差异与情境差异，都比单纯要求“请写得更有创造力”更有效。因为从项目结果看，模型对“发挥创造力”的口头提醒并不总能抵抗真正的结构性收缩。

12.4 如果想降低真实来源风险和同质化风险，该怎么筛查

从本项目经验看，后处理筛查比前置空泛警告更有用。与其在 Prompt 里反复要求“不要抄袭、尽量多样”，不如在生成后建立一套明确的质量控制流程：先做格律筛查，再做整首与句级来源扫描，最后做内部同质化比对。

尤其值得保留的几个筛查信号包括：整首字面分位是否进入历史高位、主来源覆盖率是否超过 0.2、同源句数是否在关键句位连续出现、以及同一批次样本之间是否在整词层和意象层迅速塌缩到少数骨架上。对于真正要投入展示、出版或公开使用的样本，至少应在这些维度上经过一次自动筛查和一次人工抽检。

12.5 这个项目对后续古典文本生成研究有什么启发

本项目最重要的方法启发，是不要把古典文本生成只理解为一个“审美评测”问题。只看单篇印象，会掩盖掉大量真正决定风险结构的因素；只有把任务设计、格律评估、内容分布、来源关系与内部同质化放到同一套框架里，才能真正描述模型在古典文体中的能力边界。

对未来更广泛的古典文本研究来说，这也意味着：越是传统资源池密集、文体规范清晰、局部套语高度成熟的文本类型，就越适合用“历史基线 + 结构化评分 + 批量样本比较”的方法来研究，而不能只用零散佳作去判断模型水平。

12.6 本章小结

本章的核心，不是再重复前文结论，而是把它们转成可执行的使用判断。若目标是稳定，应增加结构扶手；若目标是多样，应主动拆散单一路径；若目标是安全使用，就必须把来源筛查和同质化筛查纳入生成流程，而不能只依赖 Prompt 中的口头提醒。

换言之，本项目最终提供的不只是一个研究结论，也是一套面向后续宋词生成实践的操作框架。

第十三章 研究局限与后续工作

13.1 为什么还需要讨论研究局限

尽管本项目已经覆盖了六类任务、三家主模型和多个分析层次，但它仍然不是一个已经封闭、无可修正的最终答案。恰恰相反，越是把任务链条、数据结构与分析流程做得完整，越需要明确指出它们各自的边界在哪里。只有这样，整份报告的结论才会显得稳，而不是显得过满。

本章讨论的重点，不是推翻前文结论，而是说明这些结论成立于怎样的条件之下，又有哪些地方仍应被理解为“当前阶段最稳妥的判断”，而不是无条件可外推的终局判断。

13.2 样本分布与模型覆盖并不完全均衡

首先，本项目虽然总样本量不小，但并不是所有数据库、所有模型、所有词牌都拥有完全对称的覆盖。真正适合稳定横向比较的，主要还是三家主模型；而在某些数据库里，尤其是补充模型和探索性模型，只能作为边缘参照使用。此外，个别高压任务下也存在低样本提醒，这意味着部分细粒度差异更适合用来判断方向，而不宜被解释得过硬。

其次，样本量的“多”也不代表分布一定均匀。不同数据库的有效入库数来自真实生成与结构化成功后的结果，因此理论请求数、实际调用数和最终有效样本数并不总是完全一致。这一点不会推翻总体结论，但会影响我们对某些边缘比较的解释力度。

13.3 意象与题材标签具有后处理性质

第六章大量使用的 imagery 与 category 字段，并不是人工金标准标注，而是后处理阶段由程序批量回填得到的派生字段。这意味着它们非常适合做宏观分布、收缩方向和数据库比较，但并不等于每一首词的意象与题材都被毫无误差地“理解”了。

尤其是在跨模型、跨主题比较时，我们更应把这些字段理解为“足以支持总体判断的语义近似标签”，而不是逐首逐句都完全精确的文学分类。也正因为如此，第十章加入样本册式展示，就是为了让统计结论能够重新回到文本层接受复核。

13.4 格律引擎与来源分析也都有边界

格律评分部分的优势，是它具有统一标准和可复核性；但它的边界同样存在。多音字、变体词谱、宽严韵兼容和最佳可解释匹配，都意味着个别边界样本仍可能需要人工复核。换言之，程序分数很适合做大规模比较，却不能在所有个案上完全替代传统词学判断。

真实来源分析与同质化分析也一样。历史基线和批量扫描能够很好地识别高风险尾部，但“相似”仍不等于“借用关系已被最终定性”。尤其是在宋词这种传统共享资源极强的文类里，从高相似走到确定的借用等级判断，还需要更细的人工判读体系。

此外，如果未来继续扩展项目，还可以在两个方面进一步加强。一方面，可以引入更多模型与更长时间跨度的数据，观察模型更新后风险结构是否发生变化；另一方面，也可以把人工词学判断与当前程序评分结合起来，形成更适合古典文体生成研究的复合评估框架。

13.6 本章小结

研究局限并不削弱本项目的主要发现，反而帮助我们更准确地理解它们的适用边界：哪些结论已经足够稳，哪些地方仍需要在更多样本、更细标签和更多人工复核中继续推进。

也正因为这些边界被明确指出，本报告最后给出的整体判断才能相对扎实：大模型宋词生成已经具备相当可观的文体模仿能力，但这种能力始终伴随着形式、内容、来源和同质化四个层面的结构性代价。对这些代价看得越清楚，后续的使用与研究才越可能走得更远。

B.2 运行配置接口：任务对象如何驱动生成

C.1 附录说明

附录 C 不再重复抄录各章已经出现的大表内容，而是把 result/ 目录下可直接检索的结果表统一汇总为一份索引。它的目标是回答两个问题：第一，某一类结论对应的结果表究竟叫什么；第二，如果后续要扩写正文、补做附图或复核某个结论，应优先去看哪一个文件。

这里所说的“大表”，主要指 csv 形式的结构化结果文件；个别 md 和 json 文件虽然不是表格，但因为承担了结果说明或运行元数据的作用，也一并列入。交互式 html 图形文件这次不作为“表格索引”的主体，只在相关条目中顺带说明。result/old/ 下的文件保留为历史存档，与当前总报告的主结论不完全同步，因此在本附录中单独列出，不与现行结果混排。
